# Job Discovery and Recommendation MVP Design

Date: 2026-07-14
Status: Approved for implementation planning
Target branch: `feature/job-discovery`
Base branch: `main`

## 1. Purpose

Add a persistent job-discovery and recommendation layer to Application Cho Viego. The feature will retrieve current postings from supported free sources, normalize and deduplicate them, evaluate them against a reviewed user profile and confirmed search preferences, explain the fit, and let the user view or save a timestamped posting snapshot.

This feature does not generate a tailored resume, generate a cover letter, submit applications, or track application status. It creates the trusted discovery layer that those later workflows can consume.

The intended long-term experience is autonomous: users should eventually open the product and find refreshed recommendations waiting for them. The MVP proves the same underlying workflow through an explicit refresh action so scheduling can be added later without changing the domain or application contracts.

## 2. Approved product approach

Use a hybrid-ready architecture with an ATS-first rollout.

The first production connectors will target supported public applicant-tracking-system job feeds, beginning with Greenhouse and Lever. A curated source registry will identify supported employers and the board identifiers required by each connector. The architecture must allow additional connectors or discovery providers later, but the MVP must not depend on a paid search provider.

Automatic recommendations include only supported sources. Unsupported search results are not displayed. A user-supplied job URL is a separate intake workflow and is outside this implementation.

## 3. User experience

A user has a reviewed master profile and a separate set of job-search preferences. The product may propose preference defaults from the profile, but the user remains authoritative over career intent and confirms or edits those defaults before discovery.

The first Streamlit flow will provide:

1. Review and edit search preferences.
2. Run `Refresh recommendations`.
3. View recommendations grouped by role family and then by match level.
4. See ten results initially.
5. See a profile-fit percentage, match label, short explanation, source, verification state, and important warnings.
6. Expand a result for detailed matching reasons, material gaps, and eligibility uncertainty.
7. Open the official posting.
8. Save an immutable timestamped snapshot.
9. View saved jobs.
10. Manually check whether a saved job remains available.

The MVP does not collect dismissal reasons and does not include background scheduling.

## 4. Search preferences

Search intent is separate from career evidence. Add a `JobSearchPreferences` concept associated with a user rather than embedding it in `MasterProfile`.

The model must support:

- user identifier;
- preference version;
- one or more role families;
- user-confirmed target titles;
- semantically related title variants generated by deterministic configuration or bounded analysis;
- desired job levels, including internship, co-op, new graduate, entry level, and optionally roles requiring one to two years of experience;
- one or more location targets;
- radius where applicable;
- remote, hybrid, and on-site values classified as required, preferred, acceptable, or excluded;
- relocation approval and approved relocation regions;
- work-authorization constraints supplied by the user;
- maximum posting age, defaulting to 30 days;
- excluded companies;
- optional preferred companies;
- user career interests and role-family priority.

Profile-derived defaults must be reviewable. The system must not infer that past experience fully represents the user’s desired next role.

## 5. Source discovery and connectors

### 5.1 Source registry

Add a persistent or configuration-backed supported-source registry with records containing:

- company identifier;
- canonical company name;
- connector type;
- external board identifier or base endpoint;
- enabled state;
- supported regions when known;
- last successful refresh timestamp;
- last failure summary when applicable.

The initial registry can be curated manually. This is an intentional MVP constraint and must be documented in the UI and project documentation.

### 5.2 Connector boundary

Define a provider-neutral port for retrieving postings from one supported source. Infrastructure adapters implement Greenhouse and Lever behavior. Application services orchestrate connectors and must not depend on provider SDKs or provider-specific response shapes.

Each connector must:

- use only public endpoints allowed by the source;
- use explicit timeouts;
- apply bounded retries only for retryable network failures;
- report partial source failures without failing the entire discovery run;
- avoid logging raw user profiles or secrets;
- normalize transport data into provider-neutral discovery records;
- expose enough source metadata to construct a canonical official URL;
- support fixture-based offline tests.

The MVP must not scrape LinkedIn, Indeed, JavaScript-heavy arbitrary career pages, or anti-bot-protected pages.

## 6. Domain concepts

### 6.1 `DiscoveredJob`

A provider-independent normalized posting candidate containing:

- stable internal job identifier;
- source connector type;
- source company identifier;
- provider external job identifier;
- canonical official URL;
- company name;
- title;
- description when available;
- normalized locations;
- remote, hybrid, or on-site state when stated;
- job level when stated or classified;
- role-family candidates;
- posted date when supplied;
- requisition identifier when supplied;
- verification status;
- fetched timestamp;
- raw-content or normalized-description hash;
- completeness warnings.

### 6.2 Verification status

Use these statuses:

- `verified_active`: the supported official source returned the posting as active;
- `verified_status_unknown`: the supported source returned the posting but did not provide enough information to prove active status;
- `unverified`: reserved for future connectors and not eligible for automatic recommendation in this MVP;
- `unavailable`: a previously saved posting can no longer be retrieved;
- `expired`: the source explicitly marks the posting closed or the posting violates a reliable expiration rule.

Only supported-source postings enter automatic recommendation. A supported posting with an incomplete description may appear in a separate `Needs details` group with a provisional score and visible warning.

### 6.3 `EligibilityAssessment`

Represent eligibility independently from fit score:

- `eligible`;
- `ineligible`;
- `unknown`.

Include explicit reason codes and human-readable explanations. A hard conflict must never be represented only as a negative score.

Hard ineligibility rules include:

- expired or unavailable posting;
- wrong country or excluded location;
- excluded work arrangement;
- explicit work-authorization conflict;
- unsupported job level;
- explicit degree conflict;
- explicit graduation-date conflict;
- user-excluded company;
- experience requirement substantially above the selected level.

Unknown information does not block a recommendation and does not count as confirmed eligibility. Only important unknowns should be displayed to avoid burdening the user.

### 6.4 `JobRecommendation`

A profile-specific evaluation containing:

- recommendation identifier;
- job identifier;
- user identifier;
- profile identifier and profile version;
- search-preference version;
- eligibility assessment;
- deterministic fit score;
- match label;
- verification confidence;
- top matching reasons;
- material gaps, limited to the three most important;
- important unknowns;
- role-family section;
- generated timestamp;
- optional bounded model-assisted explanation metadata.

The percentage is a profile-fit score. It must never be described as the probability of receiving an interview or offer.

### 6.5 `SavedJob`

A saved job stores:

- user identifier;
- immutable normalized posting snapshot;
- saved timestamp;
- source URL;
- source verification state at save time;
- current availability state;
- last manual availability-check timestamp.

If the live posting disappears, retain the snapshot and mark the live source unavailable. Do not delete the saved job automatically.

### 6.6 `DiscoveryRun`

Record operational metadata for each refresh:

- run identifier;
- user and profile identifiers;
- profile and preference versions;
- start and completion timestamps;
- sources attempted;
- source warnings;
- retrieved count;
- normalized count;
- duplicate count;
- eligibility-filtered count;
- scored count;
- returned count;
- whether model assistance was used;
- model call count;
- terminal run status.

This record supports debugging, future scheduling, and cost control without storing hidden model reasoning.

## 7. Normalization and deduplication

Normalize provider data before ranking. At minimum normalize:

- company casing and aliases;
- title whitespace, punctuation, and common level markers;
- location components;
- work arrangement;
- description whitespace;
- posted timestamps;
- canonical URLs.

Deduplicate using a layered policy rather than URL alone. Prefer, in order:

1. matching provider and external job identifier;
2. matching requisition identifier and company;
3. matching canonical official URL;
4. matching normalized company, normalized title, location, and description hash or high description similarity.

When duplicates are found, prefer the official ATS record and merge only non-conflicting useful metadata. Distinct departments, requisitions, locations, or materially different descriptions remain separate jobs.

## 8. Ranking policy

### 8.1 Pipeline

The discovery pipeline is:

1. Retrieve up to a configured bounded number of candidates from supported sources.
2. Normalize records.
3. Remove duplicates.
4. Determine verification and completeness.
5. Apply hard eligibility rules.
6. Deterministically calculate fit scores for eligible and unknown candidates.
7. Keep a bounded finalist set.
8. Optionally use Gemini for ambiguous semantic classification or concise finalist explanations.
9. Revalidate any model-assisted output against deterministic facts and constraints.
10. Persist recommendations and the discovery-run summary.
11. Return ten initial recommendations grouped by role family and match level.

### 8.2 Initial deterministic weighting

Start from a 100-point score:

- 30 points: demonstrated technical evidence;
- 20 points: required skill and capability coverage;
- 15 points: career-interest and role-family alignment;
- 15 points: experience-level match;
- 10 points: education and relevant-coursework match;
- 5 points: preferred skill coverage;
- 5 points: posting recency and completeness.

Location, work authorization, and work arrangement are primarily eligibility constraints, not ordinary positive-score categories.

Initial labels:

- 85 to 100: `strong_match`;
- 70 to 84: `good_match`;
- 55 to 69: `stretch`;
- below 55: `not_recommended`.

These thresholds and weights must be defined in one focused policy module and covered by tests so they can be revised without changing connectors or UI code.

Hard-rejected jobs do not receive an ordinary user-facing match score. The UI may show an aggregate filtered-out count and expandable reasons for debugging or transparency, but rejected jobs must not be mixed into recommendations.

### 8.3 Missing descriptions

A complete description is required for a normal full-confidence score. If a supported source returns a posting with missing or materially incomplete description text:

- retain it only as a provisional discovery result;
- display a clear `Missing description` or `Needs details` warning;
- apply a documented completeness penalty;
- keep it outside normal strong/good recommendation competition;
- do not represent it as fully verified for future resume or cover-letter use.

## 9. Gemini authority and cost control

Gemini is optional and advisory.

Deterministic code owns:

- source support;
- verification;
- deduplication;
- hard eligibility;
- scoring inputs;
- scoring thresholds;
- persistence;
- recommendation ordering after validated adjustments.

Gemini may help:

- classify ambiguous job-level or role-family language;
- map semantically related requirements to reviewed evidence;
- produce concise explanations for a bounded finalist set.

Define a separate `JobFitAnalyzer` port rather than adding discovery behavior to the existing resume-specific language-model interface.

Do not call Gemini once per retrieved posting. Apply deterministic filtering first, then use one batched call or a small fixed call budget for finalists. The discovery run records the count. When Gemini is unavailable, malformed, rate-limited, or over budget, return deterministic recommendations with a reduced-analysis notice.

Model output must be typed, evidence-linked where it references candidate qualifications, and revalidated. It must not invent job requirements, user qualifications, work authorization, salary, location, dates, or source status.

## 10. Persistence and multi-user boundaries

The current local MVP may use SQLite, but repositories must preserve future multi-user boundaries.

Add separate repository ports for:

- job-search preferences;
- normalized discovered jobs or recommendation snapshots;
- saved jobs;
- discovery runs;
- supported source registry if it is not configuration-backed.

Saved jobs belong to a user. Each recommendation records the profile and profile version used to produce it. A later profile update does not silently rewrite old recommendation history.

Persistence adapters must use schema validation and explicit errors for missing or corrupt records, following the existing repository style.

## 11. Application services

Use focused application services rather than one large workflow class.

Suggested responsibilities:

- `RefreshJobRecommendationsService`: orchestrates source retrieval, normalization, deduplication, eligibility, ranking, optional analysis, persistence, and run reporting;
- `SaveJobService`: stores an immutable posting snapshot for a user;
- `CheckSavedJobAvailabilityService`: manually refreshes only availability metadata while preserving the snapshot;
- `JobSearchPreferenceService`: validates and stores confirmed user search intent.

Each service depends on ports and domain policies. No application service imports FastAPI, Streamlit, provider SDKs, or database-specific code.

## 12. API surface

Add typed FastAPI endpoints for the MVP. Exact names can follow repository conventions, but the design requires operations equivalent to:

- read and update job-search preferences;
- trigger a recommendation refresh;
- list current recommendations with grouping and pagination metadata;
- retrieve recommendation details;
- save a job;
- list saved jobs;
- manually check saved-job availability;
- list or retrieve discovery-run summaries for diagnostics.

Network-source partial failures should return successful recommendations plus structured warnings. Invalid preference input returns a validation error. A total source failure returns an explicit unavailable response and persists the failed run summary.

The API must not perform provider logic or reimplement ranking.

## 13. Streamlit delivery

Implement Streamlit only after domain, ports, application services, adapters, and API contracts have focused tests.

The page should group results by role family. Within each family, display Strong Match, Good Match, Stretch, and Needs Details sections. Each card shows:

- title;
- company;
- location and work arrangement;
- profile-fit percentage and label;
- verification state;
- posting age when known;
- two or three brief matching reasons;
- visible completeness or eligibility-unknown warnings;
- official-posting action;
- save action;
- expandable details.

Show ten results initially. Pagination or load-more is deferred unless required by the API contract tests.

## 14. Error handling

One source failure must not fail the entire refresh. The discovery run and API response include structured source warnings.

Classify connector failures at minimum as:

- timeout;
- network unavailable;
- rate limited;
- malformed provider response;
- unsupported source configuration;
- provider authentication or access failure when applicable;
- unexpected provider failure.

Avoid silently swallowing failures. Avoid exposing secrets, raw profiles, or full model prompts in logs.

A saved posting that disappears is marked unavailable while its snapshot is retained.

## 15. Testing strategy

### 15.1 Domain and policy tests

Cover:

- preference validation;
- eligibility hard gates;
- unknown eligibility behavior;
- deterministic score calculation;
- label thresholds;
- missing-description penalties;
- role-family grouping;
- top-three material-gap selection;
- stable identifiers and fingerprints.

### 15.2 Connector tests

Use recorded or hand-built fixtures without live calls for ordinary tests. Cover:

- Greenhouse normalization;
- Lever normalization;
- empty boards;
- pagination where required;
- malformed records;
- timeouts and retry classification;
- canonical URL creation;
- partial missing fields.

Live source smoke tests must be separately marked and excluded from the default test suite.

### 15.3 Deduplication tests

Cover duplicates identified by:

- provider external ID;
- requisition ID;
- canonical URL;
- normalized metadata and description hash;
- similar titles that are intentionally distinct due to location, department, or requisition.

### 15.4 Application-service tests

Use fake connectors, fake repositories, and a fake optional analyzer. Verify:

- successful multi-source refresh;
- partial source failure;
- all-source failure;
- deterministic fallback when Gemini fails;
- strict model-call budget;
- persistence of discovery-run counts;
- immutable saved snapshots;
- manual availability check;
- profile and preference version capture.

### 15.5 API and UI tests

Add focused FastAPI contract tests for validation, warnings, grouping, saving, and availability checks. Keep Streamlit logic thin and move testable transformations into ordinary functions.

Run the existing offline suite before and after each implementation batch. Live Gemini and live source tests remain opt-in.

## 16. Security, compliance, and operational constraints

- Use only allowed public source endpoints.
- Respect source terms, rate limits, and retry guidance.
- Do not scrape LinkedIn or Indeed.
- Do not attempt browser automation or bypass anti-bot measures.
- Do not log API keys, raw resumes, full job descriptions, raw model prompts, or raw model responses by default.
- Bound retrieval counts, connector concurrency, timeouts, retries, and model calls through configuration.
- Treat provider data as untrusted input and validate all normalized records.

## 17. Explicit non-goals

This implementation does not include:

- resume generation handoff;
- cover-letter generation handoff;
- automatic application submission;
- application-status tracking;
- background scheduling;
- email or notification delivery;
- arbitrary web scraping;
- LinkedIn or Indeed scraping;
- paid search-provider integration;
- dismissal-feedback learning;
- salary recommendations;
- prediction of interview or offer probability;
- automatic rewriting of user search intent from behavior.

The domain and application boundaries should allow later resume and cover-letter handoff using the saved normalized posting snapshot, but no UI action or workflow for that handoff is required now.

## 18. Acceptance criteria

The MVP is complete when the following scenario passes through automated tests and a manual demonstration:

Given a reviewed engineering profile and confirmed preferences for selected role families, job levels, locations, work arrangements, relocation choices, and authorization constraints, when the user refreshes recommendations, the system retrieves postings from configured supported Greenhouse and Lever sources, normalizes and deduplicates them, excludes explicit eligibility conflicts, assigns explained deterministic profile-fit rankings, clearly marks important unknowns and incomplete descriptions, persists the discovery run and recommendations, displays ten grouped recommendations, opens the official posting, saves an immutable timestamped snapshot, and later marks that saved posting unavailable without deleting the snapshot.

The default offline test suite must pass without live Gemini or live job-source calls. Live smoke tests must be explicitly selected.

## 19. Implementation sequencing constraints

Implementation planning must preserve this order:

1. domain models and scoring/eligibility policies;
2. repository and connector ports;
3. fake adapters and deterministic service tests;
4. Greenhouse and Lever adapters;
5. SQLite/configuration persistence;
6. application orchestration;
7. FastAPI contracts;
8. Streamlit delivery;
9. documentation and opt-in live smoke tests.

Do not begin with Streamlit and do not add background scheduling during this implementation.
